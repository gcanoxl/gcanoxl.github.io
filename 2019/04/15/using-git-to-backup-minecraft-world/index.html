使用git来备份Minecraft存档 之 理论篇

在Minecraft中，世界存档备份和管理是很重要的话题。
毕竟一个存档可能是一个人甚至一群人投入了几天几个月乃至几年精力和时间的成果，谁都不希望存档丢失。
那么就需要一个合理的存档备份和管理方案。本文将会教你如何用强大的git工具，来备份和管理Minecraft存档。
无论是单机还是服务器，这种方式都可以使用，并且非常灵活强大。

传统的存档管理方式的优点和缺点

传统的管理存档主要有以下几种方式：

使用U盘备份

U盘是最为传统的备份数据的方式之一，优点是无需网络，且备份速度比较快，但缺点也就比较多了。
首先U盘必须随身携带，容易丢失，还不能做到自动备份。而且我个人使用的mbp还没有USB接口，用U盘还得随身带个转接头，太麻烦了。

总结：
安全指数：★★★☆☆
方便指数：★★★☆☆
灵活指数：★☆☆☆☆

使用网盘备份

这里的网盘包括两类，一类是以Google Drive，百度云为代表的上传型网盘，另一类是以Dropbox为代表的同步型网盘。
上传型网盘的储存空间一般比同步型网盘要大，但需要用户手动上传下载，比较不方便。而且每次必须要整个文件上传一次，也是很浪费网络资源的。

而同步型网盘则更适用于备份Minecraft存档，首先可以全自动备份，不需要用户操心，其次能做到增量同步，节省网络资源。
并且如果你有两台以上电脑，可以在一台玩完存档，在另一台电脑上玩的时候完全无缝连接。

而缺点也很明显，首先最大的问题是，在服务器上很麻烦，因为无论是Google Drive，百度云，还是Dropbox都没有提供服务器版本的客户端，这就使得服务器上的自动备份很麻烦。

在使用Git之前，我一直便使用的这种方式备份Minecraft存档。

总结
安全指数：★★★★★
方便指数：★★★★★
灵活指数：★★★☆☆

使用同步工具备份

例如rsync等同步备份工具。这种方式要实现自动备份，通常要和shell脚本及crontab结合，但即使实现了自动备份，也无非就是和网盘备份差不多，只不过可以更方便的在服务器上使用罢了。

这种方式的缺点则在于，必须要有另一个备份机器。例如要备份服务器上的存档，那么备份方式就是同步到本地来。或者要备份本地的存档，就同步到服务器去，因为rsync只是一个增量同步工具，并不会给你增加一个储存空间来。

CanCraft第一周目便使用此种备份方式，将服务器存档定时同步到所有玩家的本地。

总结
安全指数：★★★★☆
方便指数：★★★★★
灵活指数：★★★☆☆

为什么要用Git

上面已经列举了传统的备份方式，客观的说，对于要求不高的玩家来说，已经足够使用了。
但我还是选择了Git，因为Git首先Git有以上所有工具的共同优点。其次，作为一个版本管理工具，Git有以上所有备份方式都没有的逆天功能——版本控制

假设你在MC中，正打算做个大型的建筑红石机械什么的。
你收集够了材料，做好了一切前期准备工作和测算等等，豪情万丈的开始动工了，仿佛已经看到了建筑的美轮美奂和机械的精妙绝伦。
但遗憾的是，由于你对这个工程并不熟悉，等你做了一大半，你突然的发现，因为某些之前没想到的原因，导致工程无法进行下去，必须要拆掉重新来过。
这时候假如你没有使用版本控制系统，那么你必须手动一点点拆除。而若是使用了版本控制系统，你只需要一个命令，就能使存档恢复到开工之前。

而且更重要的是，你可以用版本控制系统，在任意时间段，看到详细的存档发展过程，从一个荒凉的世界，变成一片都市家园。该多么美好啊……

如果你把你的存档发给了朋友，他也可以从过去的某个版本，开始游玩，而不用从最新的版本开始游玩。

你可能会说，各个阶段把存档复制一份单独备份也可以啊。理论上是可以，但是假设存档有500M，那你备份10份就要5G。
即使你硬盘够大，但硬盘大也不能浪费啊，我们还是得坚持朴素的劳动人民作风……

而用Git管理，即使备份了一千个版本，那么也只需要500M多一点（Git需要一点空间来储存版本信息）

分之管理的策略

我个人在使用的时候，是开了两个分支，一个是master分支，一个是daily分支。
顾名思义，daily分支使用crontab每天自动提交并同步到Github，commit信息随便填都行。
而当你完成了某个工程，或者认为这是一个需要备份的时间点，那么应该merge到master分支去。
这么做主要是因为，随着时间的推移，daily分支可能有几百上千个提交，非常难以查找和管理。
